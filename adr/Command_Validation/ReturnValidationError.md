# ADR: Двухфазная валидация команд с поддержкой Force

Дата: 2026-03-04
Статус: Accepted
Автор: InsonusK

# Контекст

В проекте используется паттерн CQRS и MediatR для обработки команд.
До сих пор валидация выполнялась в pipeline (через `FluentValidation`),
и при наличии ошибок выполнение команды прерывалось.

Однако валидация DTO может содержать предупреждения (`Warnig` , `Info` ),
которые не должны полностью блокировать выполнение операции.
Также после обновления сущности необходимо убедиться,
что финальное состояние доменной модели корректно (post-validation).

# Решение
Введена двухфазная схема валидации с поддержкой управляемого выполнения через флаг Force.
1. Все команды реализуют интерфейс IForceableCommand с полем bool Force.
2. Pre-validation (до Handler):
    - Если есть `Error` → возвращается `ValidationError`, выполнение не продолжается.
    - Если есть `Warnig`  или `Info` :
        - `Force == false` → выполнение прерывается, возвращаются предупреждения.
        - `Force == true` → выполнение продолжается.
    - Post-validation (после обновления сущности):
        - Если найдены `Error` → выполняется rollback транзакции, возвращается `ValidationError`.
        - Если найдены `Warnig`  или `Info`  → изменения сохраняются, возвращаются предупреждения.
    - Предупреждения pre-validation не учитываются после успешного обновления — приоритет имеет итоговая (post) валидация доменной сущности.

# Последствия
## Положительные
- ✅ Предсказуемая и управляемая логика выполнения команд.
- ✅ Сохраняется целостность данных (post-validation с rollback).
- ✅ Возможность интерактивного управления (UI может включать `Force`).
- ✅ Разделение ответственности: pre-validation проверяет DTO, post-validation — домен.

## Нейтральные / Требующие внимания
- ⚙️ Требуется хранить информацию о предупреждениях в Result<T> (или эквиваленте).
- ⚙️ Handler должен вызывать post-validation явно.
- ⚙️ Поведение `Force` должно быть отражено в API/документации для клиентов.

# Альтернативы, рассмотренные

| Альтернатива |	Причина отказа|
| --- | --- |
| Выполнять валидацию только в `Handler` | Нарушает SRP, смешивает слои. |
| Возвращать предупреждения через `CommandContext` |	Вносит неявные зависимости. |
| Игнорировать `Warnig`  на этапе валидации |	Потеря UX-контроля и информативности. |

# Вывод

Решение фиксирует баланс между строгостью и гибкостью валидации,
сохраняя чистую архитектуру CQRS и контролируемое поведение команд.
